# Ruinae Preset Redesign — Master Implementation Plan

## 1. Goal + Background

Ruinae ships 148 factory presets generated by `tools/ruinae_preset_generator.cpp`. Users report the bank sounds **boring and same-ish** because the old suite was built almost entirely from four template helpers (`setSynthPad`/`setSynthBass`/`setSynthLead`/`setSynthAcid`, generator lines ~285–395): nearly every preset was a saw through a Ladder/SVF low-pass into reverb, 9 of 10 oscillator engines and 10 of 13 filter types were dormant, most presets had no autonomous modulation, and a class of bugs (notably `filter.envAmount` written as a 4000 Hz value into a field that is a **semitone** amount, range −48..+48) meant filter envelopes rarely did anything. This redesign replaces every preset block with a hand-authored definition so that **each of the 148 presets has a distinct sonic identity** and the suite **collectively exercises every settable functional area** of the synth (all 10 engines, all filter types, all 7 distortions, all 5 delay types, both reverbs, all 4 harmonizer modes, the full exotic mod-source roster, the mod-matrix scale/curve/bypass axes, Bezier envelopes, the global filter, the trance gate, and the complete arp lane/mode/scale/voicing space). The concrete per-preset replacement code lives in 15 **category plan files** (Section 4); this master plan is the ordered procedure to apply, generate, test, and validate them.

## 2. Ground Rules for the Implementer (READ FIRST — NON-NEGOTIABLE)

1. **Edit exactly ONE source file:** `F:\projects\iterum\tools\ruinae_preset_generator.cpp`. Do not touch `ruinae_preset_format.h`, `plugin_ids.h`, CMake, or any other file.
2. **Locate each preset block by its `p.name` string**, never by line number (line numbers drift as edits land). Each category plan entry gives the exact search string (e.g. the block containing `p.name = "Basic Up 1/16"`).
3. **Replace the ENTIRE block verbatim.** A preset block starts at its opening `{` (the line before `PresetDef p;`) and ends at the matching `presets.push_back(...);` followed by its closing `}`. Delete that whole block and paste the category plan's **Replacement code** in its place, character-for-character. Do not merge, do not keep old lines, do not reformat.
4. **NEVER invent values.** Only paste what the category plan file contains. If a plan's code does not compile, the fix is to re-read the plan code against `F:\projects\iterum\tools\ruinae_preset_format.h` (field names, types) — not to guess a replacement value.
5. **Process ONE category plan file at a time**, in the order listed in Section 3, Step A. Apply all preset blocks in that file, then build (Step B) before moving to the next file. Do not batch multiple category files between builds.
6. **Keep the preset count at exactly 148.** Every replacement is 1-block-for-1-block. Do not add or remove `presets.push_back(...)` calls. The block being replaced and its replacement each push exactly one preset.
7. **Do not rename categories.** Category strings are fixed (see the 15 folders in Section 3, Step E). The `p.category = "..."` string in each replacement must match its plan file's category exactly.
8. **No `git commit --amend`, no push.** Committing is out of scope for this plan unless a later instruction authorizes it.

## 3. Ordered Execution Procedure

All commands are **Bash tool**, run from the repo root `F:\projects\iterum` (POSIX paths). CMake MUST be invoked by full path — the PATH `cmake` is a broken Python wrapper on this machine.

Set once per shell session (shell state does not persist between Bash calls, so prefer the full path inline or re-export as needed):
```bash
CMAKE="/c/Program Files/CMake/bin/cmake.exe"
```

### Step 0 — One-time configure (only if `build/windows-x64-release` does not already exist)
```bash
"/c/Program Files/CMake/bin/cmake.exe" --preset windows-x64-release
```
If the build dir already exists, skip this.

### Step A — Apply ONE category plan file
Open the category plan file (Section 4 index). For each numbered preset entry in that file:
1. Read the **Locate** line to get the `p.name` search string.
2. Find that block in `ruinae_preset_generator.cpp` (Grep for the exact `p.name = "..."`).
3. Replace the whole `{ ... presets.push_back(...); }` block with the entry's **Replacement code** verbatim (Ground Rule 3).

Complete every entry in the current file before building.

### Step B — Build the generator after EACH category file
```bash
"/c/Program Files/CMake/bin/cmake.exe" --build build/windows-x64-release --config Release --target ruinae_preset_generator 2>&1 | tail -40
```
- **Zero errors AND zero warnings required** before proceeding (CLAUDE.md: Build-Before-Test Discipline; all code compiles with zero warnings).
- On a compile error: re-open the failing preset's Replacement code in the category plan and check each field/setter name and type against `F:\projects\iterum\tools\ruinae_preset_format.h`. Fix the pasted code to match the header exactly. Do **not** invent a value; the plan's intent is authoritative — the error is a transcription mismatch.
- Only when this file builds clean, return to Step A for the **next** category file (order below).

**Category file order** (apply + build one at a time):
1. `plan-arp-classic.md`
2. `plan-arp-acid.md`

### Step C — After ALL 15 categories build clean: regenerate the preset files
```bash
"/c/Program Files/CMake/bin/cmake.exe" --build build/windows-x64-release --config Release --target generate_ruinae_presets 2>&1 | tail -20
```
This runs the generator, writing `.vstpreset` files into `plugins/ruinae/resources/presets/<Category>/`.

Verify the count is exactly 148:
```bash
find "F:/projects/iterum/plugins/ruinae/resources/presets" -name "*.vstpreset" | wc -l
```
Expected output: `148`. If not 148, a block was duplicated or dropped — locate the discrepancy before continuing.

### Step D — Build and run the Ruinae test suite
```bash
"/c/Program Files/CMake/bin/cmake.exe" --build build/windows-x64-release --config Release --target ruinae_tests 2>&1 | tail -20
build/windows-x64-release/bin/Release/ruinae_tests.exe 2>&1 | tail -5
```
The last line must read `All tests passed (...)` or `... | N passed | 0 failed`. **Do not grep the output; read the summary line.** If any test fails, fix the cause (a factory-preset round-trip or range test usually points at an out-of-range value in a pasted block — reconcile it against `ruinae_preset_format.h`). Per CLAUDE.md Constitution VIII: never dismiss a failure as "pre-existing"; halt and fix.

### Step E — Copy regenerated presets to the install directory
The install root is `C:\ProgramData\Krate Audio\Ruinae\` with one subfolder **per category** (verified present: `Arp Acid`, `Arp Chords`, `Arp Classic`, `Arp Euclidean`, `Arp Generative`, `Arp Modulation`, `Arp Performance`, `Arp Polymetric`, `Bass`, `Experimental`, `Leads`, `Modulation`, `Pads`, `Rhythmic`, `Textures`). This mirror-copy replaces the whole tree so renamed presets don't leave stale files behind:
```bash
rm -rf "C:/ProgramData/Krate Audio/Ruinae"/*/
cp -r "F:/projects/iterum/plugins/ruinae/resources/presets"/* "C:/ProgramData/Krate Audio/Ruinae/"
find "C:/ProgramData/Krate Audio/Ruinae" -name "*.vstpreset" | wc -l   # must print 148
```
(Presets live in `C:\ProgramData\Krate Audio\Ruinae\`, NOT inside the VST3 bundle — per project reference notes.)

### Step F — Build the plugin and run pluginval
```bash
"/c/Program Files/CMake/bin/cmake.exe" --build build/windows-x64-release --config Release 2>&1 | tail -30
```
(A failure on the post-build copy step to `C:/Program Files/Common Files/VST3/` is expected and harmless — the compile is what matters.) Then:
```bash
tools/pluginval.exe --strictness-level 5 --validate "build/windows-x64-release/VST3/Release/Ruinae.vst3" 2>&1 | tail -20
```
Must report `ALL TESTS PASSED`. This is required because plugin resources (factory presets) changed.
